# Join Support Design for Marrow

## Context

Marrow needs a join system — the core relational primitive for combining two tables on shared keys. The goal is correctness first, with a multi-algorithm design that matches the approach taken by DuckDB, ClickHouse, DataFusion, and Polars: start with a general hash join, add a sorting kernel as infrastructure, then layer in sort-merge and inequality joins.

**Key findings from prior art research:**
- Marrow has zero sorting infrastructure — no `argsort`, no ORDER BY; Mojo stdlib also has no sort primitive. A sort kernel is a prerequisite for sort-based joins.
- No GPU join implementations exist in any reference system — Marrow can lead here.
- Algorithm choice is driven by: input sortedness, join condition type (equi vs. inequality), and build-side size.

## Prior Art

### ClickHouse — Orthogonal Kind × Strictness Model

ClickHouse's biggest architectural contribution: separating *which rows appear* (JoinKind) from *how many matches are used* (JoinStrictness). These are independent axes that combine to cover all analytical join patterns.

**JoinKind** (direction / output rows): `INNER`, `LEFT`, `RIGHT`, `FULL`, `CROSS`

**JoinStrictness** (matching semantics):

| Strictness | Semantics | Rows per left key | Internal structure |
|---|---|---|---|
| `ALL` (default) | Standard SQL — cartesian product for multi-match | N × M | `RowRefList` (linked list) |
| `ANY` | First match only — no row duplication | ≤ 1 per left row | `RowRef` (single pointer) |
| `SEMI` | Existence filter — left columns only | ≤ 1 | `RowRef` |
| `ANTI` | Non-existence filter — left columns only | ≤ 1 | `RowRef` |
| `ASOF` | Nearest match on last key (inequality) | ≤ 1 | `AsofRowRefs` (sorted) |

Valid combinations:
```
ALL   × INNER / LEFT / RIGHT / FULL   ← standard SQL joins
ANY   × INNER / LEFT                  ← first-match (no cartesian product)
SEMI  × LEFT / RIGHT                  ← existence filter
ANTI  × LEFT / RIGHT                  ← non-existence filter
ASOF  × INNER / LEFT                  ← time-series nearest-match
```

Key insight: `LEFT SEMI` ≈ `LEFT ANY` semantically — both filter left by existence in right, one row per key. The difference: SEMI returns only left columns; ANY returns both sides' columns.

**6 join algorithms** with explicit user selection:
- Hash Join (generic, all join types)
- Parallel Hash Join (multi-threaded build, INNER/LEFT only)
- Grace Hash Join (disk-spilling: partition → hash each partition)
- Full Sorting Merge Join (both sides pre-sorted, simple)
- Partial Merge Join (bounded memory via disk spillover, LRU block cache)
- Direct Join (dictionary/KV lookup optimization)

### DuckDB — Internal Joins for Subquery Unnesting

DuckDB exposes additional internal join types generated by the query planner when decorrelating SQL subqueries. These are never written by users — the planner emits them:

| Type | Generated from | Semantics |
|---|---|---|
| `MARK` | `EXISTS (SELECT ...)`, `x IN (SELECT ...)` | Adds boolean `?col`: `true`=matched, `false`=no match, `NULL`=nulls present |
| `SINGLE` | `(SELECT scalar FROM ...)` | LEFT OUTER but at most 1 right row per left row (runtime error if >1) |
| `RIGHT_SEMI` | Optimizer rewrite of SEMI | Build/probe side flip for cardinality-based assignment |
| `RIGHT_ANTI` | Optimizer rewrite of ANTI | Build/probe side flip |

The **MARK → SEMI/ANTI optimization**: after generating a MARK join from `EXISTS`, the optimizer checks how the boolean marker is tested:
- marker `= true` → rewrite to SEMI (cheaper, no marker column)
- marker `= false` → rewrite to ANTI

DataFusion has equivalent `LEFT_MARK` / `RIGHT_MARK` types for the same purpose.

### DataFusion — Algorithm Selection Logic

DataFusion's join selection pipeline (`join_selection.rs`):

1. If both inputs unbounded → symmetric hash join (streaming)
2. Statistical optimization: compare `total_byte_size`; swap sides if right smaller
3. Hash join partition mode:
   - Right side ≤ threshold → `CollectLeft` (buffer build side, stream probe)
   - Else → `Partitioned` (repartition both sides by key hash)
4. Sort-merge join: implemented but **not yet used in optimizer** (still experimental)

### Polars — Adaptive Build/Probe + SIMD

- Shortest side automatically becomes the build side (`det_hash_prone_order!` macro)
- Sorted state tracked per-column; sort-merge used automatically for pre-sorted data
- SIMD-accelerated sorted merge for contiguous numeric types
- Explicit cardinality validation: `OneToOne`, `OneToMany`, `ManyToOne`, `ManyToMany`
- `JoinCoalesce`: option to merge matching key columns into one (reduces duplicate output columns)

## Algorithm Landscape

### When Each Algorithm Wins

| Algorithm | Best For | Input Requirement | Memory Complexity |
|---|---|---|---|
| Hash Join | Arbitrary equijoins, random distribution | None | O(build side) |
| Sort-Merge Join | Pre-sorted data, large equijoins, unbounded streams | Both sides sorted on key | O(max same-key group) |
| Piecewise Merge | Inequality / range joins (`a.x < b.y`) | Sorted (opposite orders for `<` vs `>`) | O(matching rows per batch) |
| Grace Hash Join | Build side > available RAM | None | Adaptive via bucket count |
| Nested Loop | Non-equi predicates, tiny tables | None | O(1) |

### Algorithm Selection Logic

```
IF join has non-equi/inequality condition:
    → Piecewise Merge Join (Phase 4, needs sorting)
ELSE IF both inputs known-sorted on key:
    → Sort-Merge Join (Phase 3, zero sort cost)
ELSE IF build side fits in memory:
    → Hash Join  ← Phase 1 (implement first)
ELSE:
    → Grace Hash Join (future: needs disk I/O)
```

## Marrow Design

### Join Type Representation

The design uses ClickHouse's orthogonal `kind × strictness` model, represented as UInt8 constants (extensible, matches existing pattern in `relations.mojo`).

**JoinKind** (direction / output rows):
```mojo
comptime JOIN_INNER:  UInt8 = 0   # matching rows only
comptime JOIN_LEFT:   UInt8 = 1   # all left + matched right (NULLs for non-match)
comptime JOIN_RIGHT:  UInt8 = 2   # all right + matched left (NULLs for non-match)
comptime JOIN_FULL:   UInt8 = 3   # all rows from both (NULLs for non-match)
comptime JOIN_SEMI:   UInt8 = 4   # filter: left columns only, existence in right
comptime JOIN_ANTI:   UInt8 = 5   # filter: left columns only, non-existence in right
comptime JOIN_CROSS:  UInt8 = 6   # cartesian product; no key columns
# Internal — generated by planner for subquery decorrelation (future):
comptime JOIN_MARK:   UInt8 = 10  # boolean marker for EXISTS/IN rewriting
comptime JOIN_SINGLE: UInt8 = 11  # scalar subquery: at-most-1 match per left row
```

**JoinStrictness** (matching behavior):
```mojo
comptime JOIN_ALL:    UInt8 = 0   # default: cartesian product for multi-match
comptime JOIN_ANY:    UInt8 = 1   # first match only; no row duplication
comptime JOIN_ASOF:   UInt8 = 2   # nearest-match on last key (future, needs sorting)
```

**JoinAlgorithm** (hint for execution planner):
```mojo
comptime JOIN_ALGO_AUTO:       UInt8 = 0  # planner picks based on inputs
comptime JOIN_ALGO_HASH:       UInt8 = 1  # force hash join
comptime JOIN_ALGO_SORT_MERGE: UInt8 = 2  # force sort-merge (needs sorted inputs)
comptime JOIN_ALGO_PIECEWISE:  UInt8 = 3  # inequality/range join (future)
comptime JOIN_ALGO_GRACE_HASH: UInt8 = 4  # out-of-memory hash join (future)
```

### Output Schema Rules

- `INNER / LEFT / RIGHT / FULL`: left fields + right fields. Duplicate column names resolved with `_right` suffix.
- `SEMI / ANTI`: left fields only (right side never materialized into output).
- `CROSS`: left fields + right fields (no key columns).

## Phase 1 — Hash Join

### `Join` Relation Node (`marrow/expr/relations.mojo`)

```mojo
comptime JOIN_NODE: UInt8 = 6

struct Join(Relation):
    var left: AnyRelation
    var right: AnyRelation
    var left_keys: List[AnyValue]   # Column exprs resolved to left schema
    var right_keys: List[AnyValue]  # Column exprs resolved to right schema
    var kind: UInt8
    var strictness: UInt8
    var algorithm: UInt8
    var schema_: Schema             # pre-computed output schema
```

`AnyRelation.join(right, left_on, right_on, how, strictness)` is the plan-building API.

### `_gather()` Kernel (`marrow/kernels/join.mojo`)

The fundamental primitive: collect elements at arbitrary indices, with `-1` producing null.

```mojo
def _gather(array: AnyArray, indices: List[Int32]) raises -> AnyArray
```

Typed overloads for `PrimitiveArray[T]`, `StringArray`; then `AnyArray` dispatch. Pattern follows `filter_()` in `filter.mojo`. This is also Arrow's `take` operation — independently useful.

### `JoinHashTable` (open-addressing with intrusive chaining)

```mojo
struct JoinHashTable(Movable):
    var _table: List[Int32]          # slot → build row index (-1 = empty)
    var _table_hashes: List[UInt64]  # parallel hash array for fast comparison
    var _chain_next: List[Int32]     # intrusive linked list for collisions
    var _capacity: Int
    var _mask: Int
    var _build_keys: List[AnyArray]  # stored for equality re-check
    var _num_rows: Int
```

**Key difference from `HashGrouper`**: `_chain_next` is an intrusive linked list enabling access to ALL build rows with the same key (needed for ALL-strictness cartesian output). `HashGrouper` stores only one group-ID per key.

**Build phase**: hash all build-side keys via `hash_()` from `hashing.mojo`; insert each row into the hash table with linear probing; chain collisions via `_chain_next`. Null keys are inserted but the probe phase rejects them (SQL null ≠ null semantics).

**Probe phase**: for each probe row, look up hash, walk the collision chain, equality-check each candidate via `_keys_match()`. Collect `IndexPairs`.

**Null semantics** (`_keys_match`): mirrors `_cols_equal_at` from `groupby.mojo` but returns `False` when any key is null on either side (SQL join semantics — inverted from groupby which treats nulls as equal).

### `IndexPairs`

```mojo
struct IndexPairs(Movable):
    var left_indices:  List[Int32]   # -1 = unmatched RIGHT/FULL build row
    var right_indices: List[Int32]   # -1 = unmatched LEFT/FULL probe row
```

**Per-kind output logic**:

| Kind | Unmatched left (probe) rows | Unmatched right (build) rows |
|---|---|---|
| INNER | drop | drop |
| LEFT | emit `(left, -1)` | drop |
| RIGHT | drop | emit `(-1, right)` |
| FULL | emit `(left, -1)` | emit `(-1, right)` |
| SEMI | drop unmatched; deduplicate to 1 per left key | drop |
| ANTI | emit unmatched left | drop |

**Per-strictness logic in the match loop**:
- `JOIN_ALL`: append one `(left, right)` pair per match → natural cartesian product
- `JOIN_ANY`: if `matched[left_row]` already set, skip → at most 1 output row per left key

### Top-level `hash_join()` API

```mojo
def hash_join(
    left: RecordBatch,
    right: RecordBatch,
    left_on: List[Int],
    right_on: List[Int],
    kind: UInt8 = JOIN_INNER,
    strictness: UInt8 = JOIN_ALL,
    ctx: Optional[DeviceContext] = None,  # GPU readiness hook
) raises -> RecordBatch
```

### `JoinProcessor` in Executor

Blocking build (CollectLeft) + streaming probe:

```mojo
struct JoinProcessor(RelationProcessor):
    var left_proc: AnyRelationProcessor   # build side
    var right_proc: AnyRelationProcessor  # probe side
    var left_key_indices: List[Int]
    var right_key_indices: List[Int]
    var kind: UInt8
    var strictness: UInt8
    var schema_: Schema
    var ctx: ExecutionContext
    var _left_batch: Optional[RecordBatch]  # materialized build side
    var _exhausted: Bool
```

`pull()`: on first call, consume entire `left_proc` into one `RecordBatch` (blocking build phase), then for each subsequent call pull one morsel from `right_proc`, probe, gather, emit output batch.

## Phase 2 — Sorting Kernel (`marrow/kernels/sort.mojo`)

Prerequisite for Sort-Merge Join, ASOF Join, IEJoin, and ORDER BY in query plans. Mojo stdlib has no sort primitive — must be built from scratch.

```mojo
# Single-column sort: returns permutation indices.
# Equivalent to numpy.argsort or Arrow's sort_indices.
def argsort(array: AnyArray, ascending: Bool = True, nulls_first: Bool = True) raises -> PrimitiveArray[int32]

# Multi-column sort: sort by (arrays[0], arrays[1], ...) with per-column direction.
# Equivalent to Arrow's sort_record_batch.
def argsort_multi(
    arrays: List[AnyArray],
    ascending: List[Bool],
    nulls_first: Bool = True,
) raises -> PrimitiveArray[int32]

# Apply a permutation to produce a sorted array. Reuses _gather().
def take(array: AnyArray, indices: PrimitiveArray[int32]) raises -> AnyArray
```

**Sort algorithm**: radix sort for fixed-width primitive types (O(N), SIMD-amenable, cache-friendly); comparison-based introsort for strings and multi-column with complex comparators.

**Relation node**:
```mojo
comptime SORT_NODE: UInt8 = 7

struct Sort(Relation):
    var input: AnyRelation
    var keys: List[AnyValue]
    var ascending: List[Bool]
    var schema_: Schema   # same schema as input
```

`SortProcessor`: collect all input batches → concat → `argsort_multi` → gather.

## Phase 3 — Sort-Merge Join

Requires Phase 2. Provides better performance than hash join when inputs are pre-sorted (e.g., time-series data, range-partitioned tables).

**Two-pointer algorithm**:
1. Both pointers start at row 0
2. `left[li] < right[ri]` → advance `li` (or emit unmatched left for LEFT/FULL)
3. `left[li] > right[ri]` → advance `ri` (or emit unmatched right for RIGHT/FULL)
4. `left[li] == right[ri]` → collect equal range on both sides; emit ALL-strictness cartesian product or ANY-strictness first match; advance past equal range

Memory: O(max same-key group) — much smaller than hash join O(build side).

```mojo
def sort_merge_join(
    left: RecordBatch,  # pre-sorted on left_on keys
    right: RecordBatch, # pre-sorted on right_on keys
    left_on: List[Int],
    right_on: List[Int],
    kind: UInt8 = JOIN_INNER,
    strictness: UInt8 = JOIN_ALL,
) raises -> RecordBatch
```

`SortMergeJoinProcessor` checks metadata sortedness → uses sort-merge if both inputs sorted, else falls back to hash join. This is the `JOIN_ALGO_AUTO` implementation.

## Phase 1b — Partition-Parallel Hash Join (shipped 2026-04-19)

### `ExecutionContext` — unified CPU + GPU dispatch

Kernels accept an `ExecutionContext`
([marrow/kernels/execution.mojo](../marrow/kernels/execution.mojo))
bundling:

- `num_threads: Int` — CPU stripe-parallel worker count (`0` = auto via
  `num_physical_cores()`, `1` = serial, `>1` = explicit)
- `device: Optional[DeviceContext]` — optional GPU device

This replaces the old `Optional[DeviceContext]` parameter plus ad-hoc
`num_threads` arguments scattered across kernels. `apply()` in
[marrow/views.mojo](../marrow/views.mojo) reads the context once and
handles CPU stripe-parallelism uniformly via `sync_parallelize` —
kernels above `apply` no longer reimplement the stripe loop themselves.
The net effect on `rapidhash` alone: ~50 lines of duplicated
histogram / scatter / slice boilerplate removed, replaced by forwarding
a single `ctx` argument.

Implicit conversion from `Optional[DeviceContext]` keeps pre-existing
call sites working without source changes; explicit factories
(`ExecutionContext.serial()`, `.parallel(num_threads)`, `.gpu(device)`)
make the call-site intent obvious where it matters.

### Kernel-level thread control

`hash_join()`, `HashJoin`, `SwissHashTable.build/probe/insert`, and
`rapidhash` all accept an `ExecutionContext`. `hash_join()` retains a
convenience `num_threads: Int` parameter (default `0` → auto) that's
forwarded to the internal context; `1` forces the serial fast path,
`>= 2` opts into radix-partitioned parallel execution. The context
threads through the entire call graph so nested calls inside parallel
regions can force serial behavior and avoid over-subscription.

### `RadixPartitioner`

Implements the previously-stubbed `Partitioner` trait. Splits rows by the
top `num_bits` of their hash (default `num_bits = 6` → 64 partitions,
matching L2-fit). The `SwissHashTable` probes on the low bits
(`h & mask`), so top/bottom split keeps the partition router independent
from the per-table probe order without double-hashing.

Parallelization (enabled when `num_threads > 1` and rows ≥ 65k):

1. **Per-thread histogram** — each worker scans its `N/T` stripe and
   accumulates `histograms[t * p + pid]`.
2. **Partition-major prefix sum** — serial pass over `T × P` counters;
   writes per-`(thread, partition)` output offsets so each worker can
   scatter to disjoint regions without atomics.
3. **Parallel scatter** — each worker scans its stripe again and writes
   to its pre-computed `(partition, thread)` slot. No contention.

Output: two shared flat buffers (`N` Int32 rows, `N` UInt64 hashes).
Each `Partition` is a zero-copy `PrimitiveArray` slice sharing the
immutable backing buffer via `ArcPointer`. One allocation per output.

### `HashJoin` serial / parallel split

`build()` / `probe()` are thin dispatchers:

```mojo
def build(mut self, left, key_indices) raises:
    if self._num_threads <= 1 or left.length < _PARALLEL_THRESHOLD:
        self.build_serial(left, key_indices)
    else:
        self.build_parallel(left, key_indices)
```

`_PARALLEL_THRESHOLD = 100_000` — below this the partitioning overhead
eclipses the parallel gain. Above it, the parallel path runs:

1. Hash the full build side once (parallel SIMD).
2. `RadixPartitioner.partition()` — per-thread histograms + parallel
   scatter.
3. `sync_parallelize` across partitions; each worker owns one partition
   end-to-end:
   - `take` on the build-side keys to gather its rows.
   - `SwissHashTable.build_hashes()` to build the per-partition table
     against pre-computed hashes (no re-hashing).

`probe_parallel` mirrors the same structure: partition the probe side,
then per partition `take` + `probe_hashes` (raw candidate pairs) +
`equal` / `filter_` for equality verification + remap partition-local
indices → original row numbering. Per-partition `IndexPairs` merge into
a single result via a direct Int32 buffer memcpy (skipping the generic
`concat(AnyArray)` dispatch). Output assembly (`_assemble`) currently
stays serial — per-column `take` for output materialization.

### Public `insert_hashes` / `build_hashes` / `probe_hashes`

`SwissHashTable` exposes the hash-level primitives as first-class
public methods. Callers that have already computed hashes (like the
partition-parallel HashJoin) use them directly — no separate
`*_with_hashes` variants. Keeps the SwissHashTable itself thread-
agnostic: the concurrency pattern is "one table per partition, one
worker per partition" rather than intra-table concurrency.

### Performance

At 10M × 10M INNER join (Apple Silicon, `num_threads=0` → auto):

| Implementation | Time | vs. Polars |
|---|---|---|
| Marrow serial (pre-parallel) | 330 ms | 3.5× slower |
| Marrow parallel | 131 ms | 1.4× slower |
| Polars | 95 ms | 1.0× |
| PyArrow | 100 ms | 1.1× slower |
| DuckDB | 120 ms | 1.3× slower |

2.4× single-thread → parallel speedup; gap vs. Polars narrowed from
3.5× to 1.4×.  All 10 performance cores utilized during build and
probe; partitioning pass is memory-bandwidth-bound on the scatter.

### Known limits / future work

- **`_assemble` parallelization** — attempted per-column `take` via
  `sync_parallelize` but hit Mojo runtime crashes on Optional[AnyArray]
  closure captures. Deferred.
- **Skew handling** — a pathologically-imbalanced key distribution will
  produce one oversized partition; secondary-hash split (subdivide the
  offending partition) is a follow-up.
- **Radix bits** — currently fixed at 6. Adaptive selection based on
  build-side size is a follow-up.
- **`PartitionedOp` abstraction** — the divide/map/gather orchestration
  in `build_parallel` / `probe_parallel` is duplicated. A reusable
  `PartitionedOp[T]` trait + `partition_apply` driver would collapse
  both call sites and be reusable for future partition-parallel kernels
  (groupby, radix sort). Mojo's current generics make this non-trivial;
  tracked as a follow-up refactor.

## Future Work

- **ASOF join**: Nearest-match on last key, exact match on other keys; needs sorted inputs + sorted row refs; Polars is best reference.
- **IEJoin / Piecewise Merge**: Inequality conditions (`a.x < b.y`). Sort order selection: `<`/`<=` → ascending, `>`/`>=` → descending. DuckDB/DataFusion patterns.
- **CROSS join**: Trivial cartesian product, no hash table.
- **Grace Hash Join**: Partition-based spill-to-disk for out-of-memory joins; needs disk I/O infrastructure.
- **MARK join**: Boolean marker for `EXISTS`/`IN` subquery decorrelation.
- **SINGLE join**: At-most-1 match for scalar subqueries.
- **Join reordering**: Adaptive build-side selection (always build on shorter side, like Polars).

## GPU Design

All kernel functions accept `Optional[DeviceContext] = None` trailing parameter (zero cost when `None`). GPU-parallelizable hot paths marked with `# TODO(gpu)`:

- **Hash build**: `hash_()` → `elementwise[target="gpu"]`
- **Hash probe**: Each probe row independent → warp-level parallel lookup
- **`_gather()`**: Trivially parallel random gather
- **Radix sort**: Well-studied GPU radix sort — very fast on GPU (CUB/thrust-style)

**GPU strategy** (mirrors `similarity.mojo` preloaded pattern): build hash table CPU-side, transfer to device, run probe on device. At scale (>100K rows) probe phase dominates and GPU wins.
